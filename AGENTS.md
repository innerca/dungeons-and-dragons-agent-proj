# AI 协作指令

## 模式
- Plan：多文件写入 / 新增删除文件 / 依赖或配置变更 / 公共 API 或数据结构变更 / 设计文档变更 / 网络或危险操作
  - 先读 `spec/coding.md`
  - 若任务涉及既有设计，再读 `spec/design.md` 和相关 `docs/design/*.md`
  - 先给规划，等用户回复“执行”
- L1：单文件、局部、低风险代码修改
  - 先读 `spec/coding.md`
  - 若任务涉及既有模块设计、公共契约、数据结构、跨模块调用、质量检查，再读相关 `docs/design/*.md`
  - 输出简析后直行
- L2：文案 / 样式 / 注释 / 只读分析 / 方案讨论
  - 直行，不写文件
- Design：仅在用户明确说“设计模式”或“进入 Design”时触发
  - 读 `spec/design.md`
  - 与用户协作产出或修订 `docs/design/*.md`
  - 不写业务代码，直到用户明确切回开发

## 铁律
1. 禁硬编码密钥、凭证、私有 URL、业务魔数；改用 env / 配置 / 常量
2. 禁 `lint --fix .`
3. Plan 未确认不写操作
4. 禁默认给出破坏性回滚或覆盖命令
5. 涉及公共 API 变更，优先兼容式演进，不直接破签
6. 设计已落文档时，开发 / 修复 / 质量检查必须同时遵守 `spec/coding.md` 与相关设计文档
7. 设计文档与需求或现状冲突时，先指出冲突并暂停，不擅自改设计
8. 同一根因连续失败 3 次即停

## 输出
- 首行必须为 `[模式: X]`
- Plan：目标 / 范围 / 风险 / 步骤 / 阻断条件，结尾提示回复“执行”
- L1：意图 / 边界 / 影响范围 / 核心方案，然后执行
- L2：单行说明后直接给结果
- Design：设计目标 / 约束 / 方案 / 权衡 / 待确认

## 审计
有代码改动时，结尾附：
- [ ] 硬编码？
- [ ] 破签？
- [ ] 新 env？
- [ ] 新依赖？
- [ ] 兼容性风险？
