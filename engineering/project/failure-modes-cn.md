## 逐项说明

### 1. 冷启动不是失败，但必须单独记

从日志看，GameServer 启动后还要完成几件事：

- 建 PostgreSQL 连接池
- 建 Redis 连接
- 初始化 ChromaDB
- 下载或校验 `bge-small-zh-v1.5`
- 加载嵌入模型权重

这意味着当前服务的热态和冷态差异很大。后面做指标对比时，不能把启动阶段样本和热启动样本直接混成一个结论。

### 2. RAG 当前是“可用但不完整”

这轮日志里 `rag_chunks` 始终大于 `0`，说明小说文本检索已经接到主链路里了。但 `entities` 集合反复返回空，说明实体知识并没有稳定参与上下文。

当前最合适的表述不是“RAG 已完成”，而是：

- 小说分块检索已接入并可用
- 实体检索还存在空结果场景
- 当前属于部分命中，不是完整知识注入

### 3. `talk_to_npc` 是目前最明确的真实 bug

日志里两次出现相同异常：

- `tool=talk_to_npc`
- `success=False`
- `asyncpg.exceptions.AmbiguousFunctionError`

从堆栈看，问题落在 `npc_relationship_service.update_relationship` 的 SQL 表达式上。这个问题的价值很高，因为它不是“模型答错了”，而是一个能被明确定位到代码和数据库表达式的真实后端 bug。

当前主链路的表现是：

- 工具失败被记录
- 请求没有直接中断
- 后续 LLM 仍然会继续生成回复

这说明系统已经具备“单个工具失败不把整轮请求打崩”的基本韧性，但也说明社交链路的状态一致性还不够可靠。

### 4. 要区分系统异常和业务拒绝

这轮日志里至少有两类失败：

- 系统异常：`talk_to_npc` 的 SQL 错误
- 业务拒绝：`accept_quest success=False`、`attack success=False`

这两类失败不能混写。前者说明代码或数据层有 bug，后者可能只是玩家状态、目标状态或前置条件不满足。

如果后面继续补文档，建议把工具失败至少分成三类：

- 参数不合法
- 业务条件不满足
- 系统执行异常

### 5. 当前告警策略已经失去分辨率

这轮 `19` 条样本全部触发 `slow_request_alert`，而阈值是 `5000ms`。这通常意味着两种可能：

- 当前链路真实偏慢，阈值没问题
- 当前链路阶段性还没优化完，统一阈值已经没有区分度

不管是哪一种，结论都一样：现在这条告警不能直接拿来表示“异常比例”，更适合当作“当前整体性能仍未达标”的提醒。

### 6. 当前最重的不是工具耗时，而是 LLM 段

从这批样本看：

- `Avg tool_ms` 只有 `5.67ms`
- `Avg llm_ms` 大约 `11.83s`
- `first_token_ms` 已经可用，但 `total_ms` 仍然很长

这说明当前系统最大的优化杠杆不是工具执行，而是：

- 上下文裁剪
- 调用轮数控制
- 模型响应长度控制
- 提前结束不必要的多轮 ReAct
