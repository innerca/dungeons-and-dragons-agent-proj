# 编码规范

## 适用
- Plan / L1 模式按需读取
- L2 无需读取
- 若任务命中设计约束，还需同时读取相关 `docs/design/*.md`

## 读取设计文档的触发条件
- 修改已设计过的模块或目录
- 涉及公共 API、数据结构、跨模块调用、异步流程
- Bug 修复可能影响既有契约、边界行为、回滚策略
- 质量检查需要核对设计意图、模块边界、兼容性

## 核心规则
- 最小改动，禁止顺手重构
- 先保证正确性，再谈优化；无数据不优化
- 命名清晰，优先可读性
- 优先早返回，减少嵌套
- 边界输入必须校验，异常不能静默吞掉
- 公共 API 禁直接改签；优先新增 + `@deprecated` + 内部转发
- 禁硬编码密钥、私有 URL、魔法数字；改用 env / 配置 / 常量
- 删除文件、导出、配置、依赖时必须显式说明

## 与设计文档的关系
- 有已接受设计时，实现、维护、修复、质量检查必须同时遵守代码规范与相关设计文档
- 文档冲突、过期或缺失时，先说明，再请求进入 Design 或补文档
- 实现缺陷但不改契约：直接修代码；必要时补充原设计文档的备注
- 设计决策要变：先进入 Design，新出修订文档，不直接覆盖旧设计

## 实施约束
- 跨模块改动优先小步提交
- 重构与行为修改尽量分离；必须混合时要说明
- 外部调用要有超时、失败、降级意识
- 新依赖先看维护状态与许可证
- 调试代码交付前清理

## 审计
- [ ] 新依赖？许可证风险？
- [ ] 改接口签名？
- [ ] 边界输入已校验？
- [ ] 新环境变量？`.env.example` 已更新？
- [ ] 硬编码配置？（URL / 密钥 / 魔法数字）
- [ ] 调试代码已清除？
- [ ] 破坏风险：低 / 中 / 高
