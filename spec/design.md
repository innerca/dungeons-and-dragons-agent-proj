# 设计规范

## 适用
- 仅在用户明确触发 `Design` 时读取
- 目标是与用户协作产出可落地的设计文档，供后续开发与维护复用

## 文档落点
- 设计文档统一放在 `docs/design/*.md`
- 推荐文件名：`<主题>.md`
- 关键修订可用：`adr-<编号>-<主题>.md`

## 何时新建或修订
- 新模块 / 新流程 / 新接口 / 新数据模型：新建设计文档
- 仅实现偏差或局部 Bug，且不改契约：不新建；按需补原文档备注
- 涉及模块边界、公共契约、数据结构、迁移路径、关键假设变化：新出修订文档，不直接重写旧设计

## 文档状态
- `Draft`：讨论中，不能作为强约束
- `Accepted`：后续开发、修复、质量检查的依据
- `Superseded`：已被后继文档替代；需写明 `Superseded-By`

## 设计输出模板
```md
# <标题>
Status: Draft | Accepted | Superseded
Owner: <可选>
Related: <可选，相关代码或文档>
Superseded-By: <可选>

## 背景
- 现状
- 问题

## 目标与约束
- 目标
- 非目标
- 关键约束

## 模块边界
| 模块 | 职责 | 接口 | 依赖 |

## 接口与数据契约
- 输入 / 输出
- 异常 / 失败策略
- 兼容性要求

## 候选方案与取舍
- 方案 A
- 方案 B
- 选型原因

## 迁移与回滚
- 灰度 / 迁移步骤
- 回滚方式

## 可观测性与验证
- 日志 / 指标 / 告警
- 测试与验收

## 风险与假设
- 风险
- 假设
```

## 设计阶段要求
- 输出必须首行标注 `[模式: Design]`
- 信息不足时先追问，不用猜关键约束
- 明确影响范围、依赖方向、故障隔离、兼容性
- 默认给“够用方案”，避免过度设计

## 交接到开发
- 设计文档被标记为 `Accepted` 后，相关开发、维护、修复、质量检查需同时遵守 `spec/coding.md` 与该文档
- 如果实现需求与已接受设计冲突，先暂停并讨论是更新设计还是按现有设计执行
