# 端侧用户画像系统：简洁可拓展落地方案

## 一、方案定位

在最短时间内交付一套**可运行、可验证、可扩展**的端侧画像系统。初期以**规则引擎为主、轻量模型为辅**，确保快速上线；架构预留扩展点，未来可平滑升级至纯 AI 推理或混合模式。

## 二、核心原则

- **先跑通，再优化**：第一阶段不追求 AI 能力全覆盖，优先验证数据采集→标签生成→业务使用的完整链路。
- **模块化与接口化**：每个组件通过接口定义，实现可插拔替换。
- **降级优先设计**：所有 AI 相关能力均需有规则兜底，保证任何时候系统可用。

## 四、第一阶段：快速落地版（2-4 周）

### 4.1 实现内容

| 模块 | 实现方式 | 扩展预留 |
| :--- | :--- | :--- |
| **行为采集** | 定义统一事件格式（JSON），提供埋点工具类 | 接口 `IEventCollector`，未来可替换为更复杂采集器 |
| **特征工程** | 硬编码模板映射表，输出自然语言描述字符串 | 接口 `IFeatureExtractor`，支持从云端加载模板配置 |
| **画像引擎** | 纯规则引擎：关键词词典 + 停留时长阈值 + 频次统计 | 接口 `IProfileEngine`，实现 `RuleEngine` 和 `AIEngine` 两种 |
| **标签存储** | SQLite 单表存储 `user_id, tag_name, weight, updated_at` | 表结构预留 `source` 字段区分规则/AI，预留 `vector` 字段 |
| **配置开关** | 本地 SharedPreferences/UserDefaults 控制是否启用 AI | 可远程更新开关 |

### 4.2 规则引擎示例

```kotlin
class RuleEngineProfile : IProfileEngine {
    private val keywordMap = mapOf(
        "篮球" to "运动_篮球",
        "奶粉" to "母婴_奶粉",
        // ...
    )
    
    override fun process(events: List<BehaviorEvent>): List<Tag> {
        val tags = mutableMapOf<String, Double>()
        events.forEach { event ->
            // 关键词匹配
            keywordMap.forEach { (kw, tag) ->
                if (event.text.contains(kw)) {
                    tags[tag] = tags.getOrDefault(tag, 0.0) + 1.0
                }
            }
            // 停留时长规则
            if (event.type == "view" && event.duration > 30) {
                tags["高参与度"] = tags.getOrDefault("高参与度", 0.0) + 0.5
            }
        }
        return tags.map { Tag(it.key, it.value) }
    }
}
```

## 五、扩展路径设计

### 5.1 从规则引擎到混合引擎

保持 `IProfileEngine` 接口不变，新增 `HybridProfileEngine` 实现：

```kotlin
class HybridProfileEngine(
    private val ruleEngine: IProfileEngine,
    private val aiEngine: IProfileEngine? // 初始可为 null
) : IProfileEngine {
    override fun process(events: List<BehaviorEvent>): List<Tag> {
        // 优先使用 AI（若可用且符合条件）
        if (aiEngine != null && shouldUseAI(events)) {
            return aiEngine.process(events)
        }
        return ruleEngine.process(events)
    }
    
    private fun shouldUseAI(events: List<BehaviorEvent>): Boolean {
        // 例如：仅对搜索词、长文本、特定事件启用 AI
        return events.any { it.type == "search" || it.text.length > 20 }
    }
}
```

### 5.2 接入轻量模型（Gemma 3 270M）

- 实现 `AIProfileEngine` 类，内部封装推理队列、模型加载、Prompt 构造。
- 通过远程配置控制 `ai_enabled` 和 `ai_ratio`（灰度比例）。
- 模型文件动态下载，不在 App 安装包内。

### 5.3 特征工程模板云端化

初期模板硬编码，后续可通过接口拉取最新模板配置，实现热更新：

```json
{
  "version": 1,
  "templates": {
    "search": "用户搜索了「{keyword}」",
    "view_product": "用户浏览了「{name}」，停留{duration}秒"
  }
}
```

### 5.4 向量化检索（未来演进）

在 `user_profile` 表预留 `embedding BLOB` 字段。当需要语义相似标签检索时，引入轻量 Embedding 模型（如 gte-small）计算向量，存储后支持近似检索。

## 七、与完整方案的衔接

| 模块 | 快速版 | 完整版升级方式 |
| :--- | :--- | :--- |
| 画像引擎 | 纯规则 | 新增 `AIProfileEngine` 实现类，通过配置切换 |
| 特征工程 | 硬编码模板 | 从本地文件/云端加载模板配置 |
| 模型部署 | 无 | 动态下载 Gemma 270M，推理框架集成 |
| 降级策略 | 规则兜底天然存在 | 增加 AI 超时/格式错误时的降级逻辑 |
| 监控 | 基础日志 | 增加性能指标采集与上报 |

## 八、总结

本简洁方案的核心思想是：**用最小的工程代价跑通价值闭环，同时用接口抽象为未来 AI 能力预留标准插槽**。业务方可以先体验到画像带来的效果提升，技术团队获得真实数据后再决策是否投入 AI 模型。
