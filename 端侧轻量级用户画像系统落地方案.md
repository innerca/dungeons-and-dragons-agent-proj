# 端侧轻量级用户画像系统落地方案

> **扩展阅读**：本文档是我在 Agent 系统设计之外的**跨领域探索**。它不直接属于 D&D Agent 项目的核心范畴，但展示了我将**同一套工程方法论**迁移至陌生领域的能力。

**为什么写这份方案？**
Agent 的长期记忆依赖用户画像。云端处理有隐私和成本问题，端侧推理是一个诱人但充满约束的方向。我好奇的是：
- 在 4GB 千元机上，一个 270M 的小模型能做到什么程度？
- 如何设计自愈机制，让端侧系统在无人维护时也不崩溃？

**方案特点**：
- **可行性版**：严格的硬件约束（功耗、内存、延迟）与成本估算。这是一份**用于决策的工程论证**。
- **简洁版**：遵循“先跑通闭环，再优化”的原则，给出了一条低风险、可演进的落地路径。

**坦诚说明**：
本方案为桌面研究与架构推演的产物，尚未在真实设备上完成全链路实测。文中的性能基线来自公开基准与合理估算。我将其开源，是希望展示我**定义技术边界、权衡工程约束**的思考过程。

**TL;DR**：
这份方案的核心价值不在于“我是否跑通过 Gemma 模型”，而在于“我是否知道在端侧部署 AI 应该关注哪些关键约束”。

## 一、总体目标

在移动应用（覆盖 4GB RAM 千元机）中，以完全本地化方式完成用户行为分析，生成结构化画像标签，用于个性化推荐。满足：

- 安装包增量 < 300MB（模型动态下载）
- 运行时内存峰值 < 500MB
- 持续任务功耗：每小时耗电 < 5%（千元机）
- 端到端推理延迟 < 1.5s
- JSON 格式合规率 ≥ 99%
- 标签准确率 ≥ 85%（相对人工基准）

## 二、核心工作流

```
行为采集 → 特征工程（模板引擎） → 推理请求串行队列 → Gemma 3 270M → JSON 标签提取 → SQLite 画像库
```

所有环节均在本地完成，数据不出设备。

## 三、模型选型与部署

| 项目 | 选择 |
| :--- | :--- |
| 模型 | Google Gemma 3 270M |
| 量化格式 | GGUF Q4_K_M |
| 模型文件大小 | ~241 MB |
| 运行时内存 | ~300-400 MB |
| 上下文窗口 | 8192 tokens |

**推理框架**：
- Android：MNN-LLM（优先使用高通 QNN NPU）
- iOS：llama.cpp + Metal

**跨平台抽象**：C++ 统一接口 `LLMBridge`，双平台通过 JNI/ObjC++ 调用。

## 四、架构设计

### 4.1 模块分层

- **采集层**：埋点 SDK 输出结构化行为事件（JSON）
- **特征工程层**：模板引擎将事件转为自然语言描述
- **推理服务层**：单例串行队列 + 推理引擎
- **存储层**：SQLite 存储标签、行为日志

### 4.2 推理串行队列（解决并发冲突）

所有推理请求提交至 `InferenceQueue`（优先级队列，FIFO，支持去重）。单工作线程串行执行，确保引擎不可重入场景下的安全。

### 4.3 主动内存管理

推理前检查可用内存，若 < 800MB（千元机阈值）则降级规则引擎，避免 OOM。

## 五、关键模块实现要点

### 5.1 特征工程模板

行为事件 JSON → 模板引擎 → 自然语言描述。

示例：
- 事件：`{"type":"view_product","name":"篮球鞋","duration":45,"price":499}`
- 输出：`"用户浏览了「篮球鞋」商品页，停留45秒，价格499元"`

模板支持云端动态下发，失败回退内置版本。

### 5.2 Prompt 设计

```
System: 你是一个用户行为分析助手。根据行为描述提取标签，严格按以下 JSON 输出，不含其他内容。
{
  "interests": ["标签1", "标签2"],
  "intent": "浏览/比价/购买/无",
  "price_sensitivity": "高/中/低",
  "confidence": 0.0-1.0
}
User: {特征工程生成的描述}
```

推理参数：`temperature=0.0`，`max_tokens=256`，`stop=["}"]`。

### 5.3 健壮 JSON 提取器

- 先尝试直接解析
- 失败则正则提取第一个 `{...}`
- 移除 Markdown 标记（```json 等）
- 修复尾部逗号等常见错误
- 仍失败则记录，单次丢弃，连续失败触发熔断

### 5.4 标签存储与聚合

SQLite 表：
- `user_tags`：原始推理结果（含置信度、时间戳）
- `user_profile`：聚合画像（按标签名加权聚合，时间衰减）

## 六、降级与自愈机制（人工不可介入下的设计）

### 6.1 降级层级

| 层级 | 状态 | 行为 |
| :--- | :--- | :--- |
| L0 | AI 正常 | 完整推理生成标签 |
| L1 | AI 不可用 | 规则引擎兜底（关键词匹配） |
| L2 | 规则引擎亦受限 | 仅缓存行为数据，等待恢复后补分析 |
| L3 | 存储/内存严重不足 | 完全静默，不采集不分析 |

### 6.2 关键故障自愈

| 故障 | 自愈动作 |
| :--- | :--- |
| 模型文件缺失/损坏 | 回退 L1，后台静默重下载，完成后自动回升 L0 |
| 推理超时（>5s） | 丢弃本次，重置引擎；连续 3 次超时触发熔断（24h L1） |
| JSON 连续解析失败 | 同超时熔断机制 |
| 设备过热 | 暂停推理，释放模型；温度恢复后自动继续 |
| 存储空间不足 | 自动清理 30 天前日志及低置信度标签；仍不足降级 L3 |
| 后台任务被系统杀死 | WorkManager 指数退避重试，进度持久化断点续传 |

### 6.3 用户无感原则

任何降级均**不显示技术错误提示**。用户仅感知推荐内容精确度的轻微变化，不影响 App 核心功能。

## 七、监控与远程调控（有限人工替代）

- 客户端匿名上报：推理失败率、超时率、降级率、机型/系统版本
- 云端配置下发：动态调整超时阈值、熔断参数、强制降级开关、特定机型禁用列表

## 九、实施路径

### 阶段一：PoC（2-3 周）
- 在目标千元机实测 Gemma 3 270M 加载时间、推理延迟、内存占用、耗电
- 用 500 条真实行为数据验证标签准确率和 JSON 提取成功率
- 决策点：若准确率 < 75% 或延迟 > 3s，暂停并重评方案

### 阶段二：MVP 开发（3 个月）
- 团队配置：端侧 AI 工程师×1、Android 高级×1、iOS 高级×1、后端×1、QA×1
- 交付双端完整功能（采集→特征工程→推理→存储→降级链路）
- 实现成本：约 54-86 万元（人力+基础设施+测试设备）

### 阶段三：灰度与优化（2-3 个月）
- 1%-5% 用户灰度，监控功耗、崩溃率、标签准确率
- 持续迭代 Prompt 和特征模板
- 维护团队缩减至 1.5 人年（年成本约 67-103 万元）
