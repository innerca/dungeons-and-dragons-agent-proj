## 一、可测试性三原则（架构硬约束）

1. **依赖倒置原则 [R1][R4]**  
   涉及IO、网络、时间、随机数、第三方SDK的调用，必须通过构造函数或方法参数注入抽象接口。  
   **禁止**在业务方法内直接调用 `DateTime.Now`、`File.Read`、`Guid.NewGuid` 等静态副作用方法。

2. **副作用隔离原则**  
   纯计算函数（如折扣计算、积分换算）必须抽离为独立、无外部依赖的纯函数。

3. **防呆设计原则**  
   涉及金额、库存扣减、状态变更的接口，必须实现幂等性（如基于Token或唯一业务键去重）。

## 二、单元测试规范

- 测试结构遵循 **AAA模式**（Arrange - Act - Assert）。
- 测试命名遵循 **Given_When_Then** 语义化风格，禁止 `test1` 式命名。
- 必须包含边界测试：`null`、空集合、`0`、负数、超长字符串。
- 必须使用 `assertThrows` 验证预期业务异常。
- **只Mock边界接口**（数据库仓储、HTTP客户端、消息队列、系统时钟）。严禁Mock值对象、集合类、第三方库内部类。
- 禁止为了覆盖率编写无断言或 `assertTrue(true)` 的僵尸测试。

### 2.1 测试-源码一致性原则（🔴 强制）
- 测试失败时，**必须优先排查源码bug**，禁止通过以下方式让测试"勉强通过"：
  - 削弱断言（如将 `== 50` 改为 `>= 30`）
  - 补充不完整的Mock数据（缺少关键字段导致KeyError）
  - 添加无意义的 `or result.success` 容错逻辑
- **判定标准**：若修改测试后源码行为改变，说明测试未真实验证业务逻辑

### 2.2 反模式检测清单
AI应扫描以下测试反模式：
- [ ] 测试包含 `skip`/`skipif` 但无明确触发条件
- [ ] Mock对象被定义但未在断言中使用
- [ ] 断言使用 `>=`/`<=` 验证确定性计算（应使用 `==` 或区间断言）
- [ ] 多个测试验证同一逻辑路径（重复测试）

> **自动化检测建议**：CI可集成 `grep -r 'assertTrue(true)' src/test` 扫描，发现即阻断。

## 三、代码审查量化红线

**满足以下阈值将触发CI质量门禁失败，MR禁止合并，需重构后重新提交。**

| 检查项 | 量化阈值 | 触发动作 |
| :--- | :--- | :--- |
| 分支语句密度 | 单个方法内 `if` / `while` / `for` / `foreach` / `case` 关键字总数 **> 15个** | 🟡 复杂度过高，强制重构 |
| 复合条件复杂度 | 单个 `if` 语句中 `&&` 或 `\|\|` 出现次数 **> 3次** | 🟡 逻辑耦合过深，建议抽取方法 |
| 方法长度 | 非空行、非注释行 **> 80行** | 🔵 建议拆分 |

> **豁免说明**：若因业务状态机等合理场景导致分支数超标，可在代码注释中说明，经Tech Lead确认后标记 `@review-exempt` 豁免。

## 四、数据库事务边界红线 [R5]

- **[R5] 禁止在数据库事务边界内进行RPC、HTTP、消息队列发送等非数据库IO操作。**
- 事务方法内仅应包含数据库操作。外部调用必须在事务开启前或提交后执行。
- 违规示例：`@Transactional` 方法内调用 `httpClient.post()` 或 `kafkaTemplate.send()`。
- 违规示例：Java: `@Transactional` 方法内调用外部API
- 违规示例：Python: `@transaction.atomic` 函数内发送HTTP请求
- 违规示例：Go: `db.Begin()` 后调用 `http.Post()`

---

## 五、Stream / Lambda 异常处理硬性检查

- 在 `Stream` / `Lambda` 表达式的 `map`、`forEach`、`peek` 等操作中，**必须显式捕获受检异常**。
- 禁止在Lambda中抛出未经包装的 `RuntimeException` 导致流处理静默中断。
- 违规示例：`.map(item -> { /* 可能抛异常的代码，无try-catch */ })`

---

## 六、依赖注入生命周期安全（🟡 主要缺陷）

- `Scoped` 服务（如 `DbContext`、用户上下文相关对象）**严禁**注入到 `Singleton` 服务中。
- 此项为 🟡 **主要缺陷**，必须修复，不可仅作提示忽略。

---

## 七、消息队列中间件硬性阻断

- 若代码包含 `@KafkaListener` / `@RabbitListener` / `@SqsListener` 等消息监听注解，**必须**显式配置 `errorHandler` 或在 `catch` 块中调用 `ack.acknowledge()`。
- 缺失此项视为 🟡 **主要缺陷**，必须修复。

---

## 八、红线豁免标记机制（含防滥用条款）

开发者可在代码注释中使用以下标记申请豁免：

```java
// @review-exempt: <规则编号> - <明确理由>
```

**有效理由示例**：
- `// @review-exempt: R1 - 第三方库强制要求调用静态方法`
- `// @review-exempt: R4 - 与遗留系统交互必须使用 DateTime.Now`

**审查人处理流程**：
1. 发现豁免标记 → 跳过对应规则的自动化阻断。
2. 人工评估理由：成立且不影响安全与正确性 → 转为 🔵 建议备注放行；不成立或存在风险 → 升格为 🟡 主要缺陷。

**防滥用条款**：
- **若单个PR中包含3处及以上的 `@review-exempt` 豁免标记，必须由架构师或Tech Lead二次确认，AI审查自动挂起，转为人工深度审查。**

## 九、安全红线详细定义

### R6：禁止SQL字符串拼接与动态查询注入

**违规判定**：使用 `+` 或字符串模板拼接用户输入构造SQL；ORM中拼接动态条件；NoSQL中使用 `$where` 拼接。

**违规示例**：
```java
String sql = "SELECT * FROM users WHERE username = '" + username + "'";
```

**合规替代**：参数化查询、ORM参数绑定、动态排序字段使用白名单校验。

### R7：禁止敏感信息写入日志

**违规判定**：日志中包含密码、Token、密钥、身份证号、银行卡号；打印完整SQL含敏感值。

**违规示例**：
```java
log.info("用户登录：username={}, password={}", username, password);
```

**合规替代**：不记录敏感字段或进行脱敏处理。

### R8：禁止缺失服务端访问控制校验

**违规判定**：接口未校验当前用户是否有权限操作指定资源（水平越权/垂直越权）；仅前端隐藏按钮。

**违规示例**：
```java
@GetMapping("/user/{userId}/profile")
public Profile getProfile(@PathVariable String userId) {
    return profileService.getById(userId);  // 未校验userId是否属于当前用户
}
```

**合规替代**：校验资源归属，使用 `@PreAuthorize` 注解。

### R9：禁止服务端请求伪造（SSRF）风险

**违规判定**：直接将用户提供的URL传递给HTTP客户端；允许 `file://`、`dict://` 等危险协议；未限制内网IP。

**合规替代**：协议/域名白名单 + 内网IP过滤 + DNS重绑定防护。

### R10：禁止不安全反序列化

**违规判定**：从不受信任数据源反序列化且无类白名单；Jackson `enableDefaultTyping`；动态反射实例化用户输入的类名。

**合规替代**：配置类白名单过滤器；使用安全序列化格式（JSON禁用多态、Protobuf）。

### R11：禁止命令注入与路径穿越

**违规判定**：用户输入拼接到系统命令；文件路径未规范化校验（`../` 穿越）；文件上传未校验扩展名。

**合规替代**：参数化命令调用；路径规范化并校验基础目录；文件扩展名白名单。

### 安全红线速查表

| 编号 | 快速搜索词 |
| :--- | :--- |
| R6 | `+ "SELECT`、`+ "UPDATE`、`$where` |
| R7 | `log.*password`、`printStackTrace()` |
| R8 | `@GetMapping.*/{.*Id}` + 无权限校验 |
| R9 | `URI.create(`、`HttpClient.*url`、`file://` |
| R10 | `ObjectInputStream`、`enableDefaultTyping` |
| R11 | `Runtime.exec`、`ProcessBuilder`、`../` |

## 十、禁止行为总览

- **[R1]** 方法内部 `new HttpClient()`、`new DbConnection()` 等关键依赖。
- **[R2]** `catch (Exception e) { }` 空处理或仅打印堆栈。
- **[R4]** 直接调用 `DateTime.Now` / `File.Read` / `Guid.NewGuid` 而未注入抽象。
- **[R5]** `@Transactional` 方法内调用外部API或发送消息。
- **[R6]** SQL/NoSQL字符串拼接。
- **[R7]** 敏感信息写入日志。
- **[R8]** 接口缺失资源归属校验。
- **[R9]** 直接使用用户输入的URL发起请求。
- **[R10]** 无白名单反序列化用户输入。
- **[R11]** 命令拼接或路径未校验。
- `Scoped` 服务注入到 `Singleton` 服务中。
- Lambda表达式内未捕获异常导致流处理中断。
- 在循环体或高频切面中新增 `log.info` 级别日志。

## 附录 A：AI规则检测算法（伪代码参考）

### A.1 分支密度检测（规则：单个方法分支数 ≤ 15）

**检测步骤**：
1. 使用AST解析源代码，定位所有函数/方法定义节点。
2. 遍历函数体，统计以下关键字出现次数：`if`、`while`、`for`、`foreach`、`case`。
3. 若计数 > 15，标记违规位置：`文件路径:行号:方法名`。

**伪代码**：
```python
def check_branch_density(ast_node):
    count = 0
    keywords = ['if', 'while', 'for', 'foreach', 'case']
    for child in ast_node.walk():
        if child.type in keywords:
            count += 1
    return count <= 15
```

### A.2 复合条件复杂度检测（规则：单个if内 &&/|| ≤ 3）

**检测步骤**：
1. 定位所有 `if` 语句的条件表达式。
2. 统计表达式树中逻辑运算符 `&&` 和 `||` 的数量。
3. 若计数 > 3，标记违规位置。

**伪代码**：
```python
def check_condition_complexity(if_node):
    count = 0
    for token in if_node.condition.tokens:
        if token.type in ['&&', '||']:
            count += 1
    return count <= 3
```

### A.3 红线规则 R1-R11 检测映射表

| 规则编号 | 检测方式 | 关键词/模式 |
| :--- | :--- | :--- |
| R1 | AST查找方法内 `new` 表达式 | `new HttpClient(`、`new DbConnection(` |
| R2 | AST查找空 `catch` 块 | `catch (` 后紧跟 `{ }` 或无 `throw`/`log` |
| R4 | AST查找静态调用 | `DateTime.Now`、`File.Read`、`Guid.NewGuid` |
| R5 | AST查找事务注解内的方法调用 | `@Transactional` + 内部 `httpClient.`/`kafkaTemplate.` |
| R6 | 正则匹配字符串拼接SQL | `+ "SELECT`、`+ "UPDATE`、`+ "DELETE` |
| R7 | 正则匹配日志语句含敏感词 | `log.*password`、`log.*token`、`log.*ssn` |
| R8 | AST查找Controller方法无权限注解 | `@GetMapping` 无 `@PreAuthorize` 且参数含资源ID |

## 附录 C：AI审查上下文理解原则

### 原则1：先理解意图，再判定违规

AI在发现以下模式时，不应直接判定违规，而应进行二次判断：

| 发现模式 | 二次判断逻辑 | 结论 |
| :--- | :--- | :--- |
| `new HttpClient()` | 是否在工厂方法或 `@Configuration` 类中？ | 若是 → 合规；若在业务方法内 → 违规 |
| `catch (Exception e) { }` | 上层是否有全局异常处理器？或此层是否为重试中间层？ | 若有全局处理器且此处仅做日志 → 合规；否则违规 |
| `DateTime.Now` | 是否在 `ISystemClock` 实现类内部？ | 若是 → 合规（防腐层内部允许）；否则违规 |

### 原则2：区分“规范违规”与“设计缺陷”

- **规范违规**：明确违反R1-R11中任一条，且无有效豁免标记 → **自动阻断**。
- **设计缺陷**：代码坏味道（如方法过长、参数过多、圈复杂度高）但未触及红线 → 标记为 **🔵 建议**，不阻断合并。

### 原则3：识别并审计豁免标记

1. 扫描代码注释中的 `@review-exempt: <规则编号> - <理由>`。
2. 提取规则编号和理由文本。
3. 判断理由合理性：
   - 若涉及第三方库强制要求、遗留系统交互 → 降级为 🔵 建议。
   - 若理由空洞（如“懒得改”）或无理由 → 保持原违规等级。
4. 将评估结论写入输出的 `exempt_audit` 字段。

### 原则4：豁免标记数量告警

- 若单个PR中豁免标记数量 ≥ 3，AI应在输出中增加 `warning` 字段：
  ```json
  "warning": "本次PR包含3处豁免标记，已自动挂起，请架构师人工确认"
  ```
- CI收到此警告后，应阻止自动合并，等待人工审批。

- **红线匹配**：将扫描结果映射到R1-R11规则编号。

### D.3 度量数据获取优先级

1. 优先使用CI环境变量中已生成的报告路径。
2. 若报告不存在，AI应仅执行基于AST和正则的本地扫描，并将覆盖率字段标记为 `"待扫描"`。

## 0. 审查前置说明与豁免机制

### 0.1 自动化规则触发原则
本报告所列红线与主要缺陷，依据静态代码分析工具及人工审查规则判定。

### 0.2 豁免标记 (Override Tag)
若代码存在特殊场景确需打破规则，开发者可使用注释申请豁免：

```
// @review-exempt: <规则编号> - <明确理由>
```

**审查人处理流程**：
1. 发现豁免标记 → 跳过对应规则的自动化阻断。
2. 人工评估理由：成立且不影响安全与正确性 → 转为 🔵 建议备注放行；不成立或存在风险 → 升格为 🟡 主要缺陷。

**防滥用条款**：
- 若单个PR包含 **3处及以上** 豁免标记，必须由架构师或Tech Lead二次确认，AI审查自动挂起。

**红线规则编号对照表**：

| 编号 | 规则内容 |
| :--- | :--- |
| R1 | 禁止在业务方法内 `new` 关键依赖 |
| R2 | 禁止空 `catch` 块 |
| R3 | 核心计算逻辑零单测 |
| R4 | 直接调用静态副作用方法（时间、文件、随机数） |
| R5 | 禁止在数据库事务内进行RPC/HTTP/消息发送等非数据库IO |
| R6 | 禁止SQL字符串拼接与动态查询注入 |
| R7 | 禁止敏感信息写入日志 |
| R8 | 禁止缺失服务端访问控制校验（越权风险） |
| R9 | 禁止服务端请求伪造（SSRF）风险 |
| R10 | 禁止不安全反序列化 |
| R11 | 禁止命令注入与路径穿越 |
